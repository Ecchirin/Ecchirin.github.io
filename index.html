Case study 01  ·  Affyn  ·  2022

A bridge between a centralised game economy and a decentralised token.

Problem

Affyn ran a virtual in-game currency that players earned through gameplay. The token, on a public blockchain, was the financial layer. We needed to let players convert one into the other — safely, predictably, and at scale, across two systems with very different consistency models. The game server is a fast, mutable, centralised database. The blockchain is slow, immutable, and finalised by consensus. They disagree about what "now" means.

Approach

The conversion contract held the entire bridge in a single state machine with explicit phases: locked, requested, confirmed, settled. Players initiated a conversion against a balance that was atomically locked on the game side; the contract minted only after a confirmation window had passed and a reconciliation job had verified that the game database, the request log, and the onchain event stream all agreed on the amount. Roles were tightly scoped — the minter key could only mint, never withdraw; admin operations moved through a Gnosis Safe multisig.

Daily minting caps acted as a circuit breaker. If anomalous volume tripped the threshold — player exploit, server bug, anything — the contract paused and escalated to manual review before a single token left the door.

CertiK-cleared. Double-spend prevention. Daily circuit breakers. Zero post-launch incidents on the conversion path.

What was hard

Distributed consistency under partial failure. The game server can crash mid-conversion. The blockchain can reorg shallow blocks. The reconciliation job can be late. Each combination needed a defined recovery path that didn't leak tokens, double-mint, or leave a player stuck. The eventual design treated every conversion as idempotent against a unique request ID and settled in only one direction — if anything looked wrong, refund and roll back on the game side, never mint to "fix" it.

Every state had to survive the question: "What if this is the only message that gets through?"

Reflection

If I built it again I'd make the reconciliation job push state instead of pull — have the game server emit confirmations onchain rather than the contract asking the game server "did this happen?". Same guarantees, less coupling, and no hidden assumption that the game side is always reachable when the chain wants an answer.

Case study 02  ·  Affyn  ·  2022

9.8 billion NFT metadata entries, indexed by geography.

Problem

The product needed an NFT for every Uber H3 hexagonal cell on the planet at a fine resolution — nearly ten billion entries, each with rich metadata, each individually queryable by location. A naive document store would have eaten storage in volumes that would have made the unit economics unworkable. We needed the same data, accessible the same way, in a fraction of the space.

Approach

Three things did most of the work. First, aggressive MongoDB schema compression: every field name shrunk to one or two characters, every enum encoded as a small integer, every default omitted from the document. Single documents that would have been 800 bytes naive came in at well under 100.

Second, geospatial indexes. H3 cell IDs were stored as the primary key, with secondary indexes on parent cells at coarser resolutions to make hierarchical queries cheap. Looking up "all cells inside this region" was a bounded index scan, not a scan-and-filter.

Third, deliberate document layout. Hot fields lived at the front; cold fields were nested. Queries that only needed coordinates and ownership never paid the cost of reading the full document.

9.8 billion entries. Roughly an order of magnitude less storage than a naive implementation. Hierarchical geospatial queries in milliseconds.

What was hard

Generating the data was a multi-day pipeline that had to be resumable, idempotent, and verifiable. We couldn't afford to start over if a worker crashed eight hours in. The pipeline ran in shards keyed by H3 parent cell, each shard checkpointed its progress to a durable log, and the orchestrator could pick up exactly where a failed shard left off. Verification was a separate pass that sampled cells at each resolution level and asserted invariants — if a parent cell claimed seven children, all seven existed; if a region had a population estimate, the sum of its children's estimates matched within tolerance.

Reflection

Schema compression is a one-way door. Once you're three months in and tooling has been written against single-character field names, you can't widen them without a migration that touches every document. If I started again I'd codify the field-name map in a generated header somewhere both the writer and the reader compile against, so a rename means a build break, not a runtime mystery.

Case study 03  ·  IGG Singapore  ·  2020–2022

Multithreaded C++ on a live game server, before the Web3 detour.

Problem

Live games don't tolerate latency, dropped messages, or inconsistent state — every player is the worst critic, and the failure mode is "they uninstall." My job was building gameplay systems on top of IGG's production C++ game-server framework: high concurrency, long-lived sessions, MySQL-backed persistence, Protobuf over the wire to clients. The whole thing ran multithreaded by default, which means every state change crosses thread boundaries unless you're deliberate.

Approach

Two disciplines mattered more than anything else: ownership and message-passing. Game state was owned by exactly one thread per shard; cross-shard interactions went through queued messages, not shared memory. Read locks were rare; write locks were a code smell. Where lock-free patterns made sense, we used them; where they didn't, we went with the simplest mutex and measured.

Protobuf schemas were a typed contract between server and client. Adding a field meant updating the schema first, regenerating both sides, and only then writing the logic. This sounds slow; in practice it caught a long tail of off-by-one and type-confusion bugs at compile time that would have been ugly to debug live.

The MySQL layer needed serious indexing work. Player queries, leaderboards, and inventory lookups all had different access patterns; a single composite index didn't serve all of them. I ended up rewriting a fair chunk of the schema and queries to match how the game actually read data, not how it would have been structured if we were starting from scratch.

Live players. Multithreaded C++. Production game-server framework. Thread-safety as a daily discipline, not a final code review.

What was hard

Reasoning about thread interactions doesn't get easier the more code you write — it gets easier the more you constrain yourself. The systems that survived were the ones with the smallest amount of shared mutable state. The bugs that bit hardest were always the ones where I'd convinced myself "this part doesn't need a lock because it's only ever called from this thread" — and then six months later a feature reused the helper from a different thread.

Reflection

A lot of the discipline I bring to smart contracts now — explicit state machines, minimal shared mutable state, treating every interaction as something that might fail — came from this period. The blockchain just enforced the discipline more strictly because the cost of getting it wrong was paid in tokens instead of crash dumps.
